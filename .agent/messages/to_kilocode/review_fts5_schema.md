# 任务交接指引 (To: Kilo Code / GLM-5)

> 发起方: Gemini 3.1 Pro (Antigravity)  
> 时间: 2026-03-07  

## 背景说明

Luvia-Gallery 项目最初使用了 `sql.js` (WebAssembly) 作为主程序的 SQLite 存储引擎，导致在点赞、修改、增加文件等操作时会**阻塞 Node.js 主线程**进行全量数百兆的文件回写 `fs.writeFileSync()`。

在上一阶段我分析了搜索功能的实现，原计划采用简单的 `LIKE` 模糊查询。但用户明确指出实际生产环境的存储量已高达 **90万条数据**，这让 `sql.js` 原有 `LIKE` 会直接导致 Node.js 崩溃式延迟，并且原包不支持原生 `FTS5`。

## 移交诉求

我已经撰写了全新的架构替换报告，计划废除低效写放大的 `sql.js` 转为无缝替代方案：基于原生绑定的 `better-sqlite3`。它在底层不仅天然支持 WAL 并发日志模式，亦支持基于表级触发器的 `FTS5` 实装。

你作为系统的架构校验中枢，此重构复杂度较深，具有多端连带风险，请你接手当前会话：

1. **阅读交接文档:** `SESSION_HANDOVER.md`
2. **阅读设计架构稿:** `.agent/docs/plans/ARCH_FTS5_SEARCH_PLAN.md`
3. 执行我在上述文档第三部分 `@重点审计请求` 所描述的三点系统风险推演 (特别是不同端安装含有 C++ 原生构建产物的扩展包的兼容风险)。
4. 如果审计通过或者您修改打磨了更安全的方案，请直接接手指引用户进行本次重构及全端联调功能验收。
